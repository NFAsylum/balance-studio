## Task: implementar `LocalDesigner/LocalJudge/LocalIterator` (backend LLM local via llama-server)

**Contexto:** já testamos a conexão do container com o servidor local. Endpoint OpenAI-compatible em `http://192.168.3.92:8080/v1`. Modelo carregado: Qwen2.5-Coder-7B-Instruct Q4_K_M, GPU offloaded, ~100 tokens/s. Configuração em `.env` (já criado com `LLM_BACKEND=local`).

Isso **substitui** o plano original de usar Anthropic no Sprint 6. Não precisa mais de API key paga.

### Ação

1. **Dependência:**
   ```bash
   poetry add openai
   ```

2. **Novo arquivo `core/llm_local.py`** com três classes implementando os Protocols de `core/llm_hats.py`:

   **`LocalDesigner` implementa `DesignerLlm`:**
   - Prompt inclui: brief, schema (via `EntitySchema.to_llm_schema()`), constraints como texto, n desejado
   - Estratégia de saída estruturada — **NÃO confiar em `tools` param puro** com esse modelo pequeno. Padrão recomendado:
     1. Prompt pede JSON explícito seguindo o schema
     2. Enviar com `response_format={"type": "json_object"}` (llama-server suporta)
     3. Parse do JSON, validar contra `EntitySchema.build_model()` gerado
     4. Se falha de schema, retry com feedback do erro no prompt (max 3 retries)
     5. Filtrar entidades que violam constraints pós-geração
   - Retornar `list[Entity]` válidas + logar as descartadas

   **`LocalJudge` implementa `SubjectiveJudgeLlm`:**
   - Prompt em `core/prompts/judge_<criterion>.txt` — critérios: `variety`, `cohesion`, `thematic_consistency`
   - Pede resposta em JSON: `{"score": 0.0-1.0, "rationale": "..."}`
   - Retorna `JudgeResult`

   **`LocalIterator` implementa `IteratorLlm`:**
   - Prompt inclui: entities atuais + sim_metrics + judge_metrics + objectives + últimas 5 mudanças por autor
   - Pede JSON: `{"modifications": [{kind, target, payload, reasoning}]}`
   - Nunca propor mudança em entity cujo último edit tenha `actor="user"` — filtrar output se propuser
   - Retorna `list[Modification]`

3. **Cliente OpenAI compartilhado:**
   - Wrapper interno em `core/llm_local.py` com `_get_client()` que retorna `openai.OpenAI(base_url=..., api_key="local")` (api_key literal "local" — llama-server ignora)
   - Timeout 180s (Iterator pode ter prompt longo)
   - Sem retry exponencial complexo — servidor local não tem 429

4. **Factory (extensão de `core/llm_hats.py` ou novo `core/llm_factory.py`):**
   ```python
   LLM_BACKEND=fake     -> Fake{Designer, Judge, Iterator}
   LLM_BACKEND=local    -> Local{Designer, Judge, Iterator}
   LLM_BACKEND=anthropic -> [não implementar agora — deixa NotImplementedError]
   ```

5. **Smoke test `scripts/smoke_local.py`:**
   - Verifica endpoint `/v1/models`
   - Roda 1 chamada de Designer (2 entidades card_game)
   - Roda 1 Judge (variety em uma lista de 3 cards do seed)
   - Roda 1 Iterator (com fake sim_metrics)
   - Imprime latência por chamada

6. **Testes automatizados:**
   - `tests/test_llm_local.py` — 6 testes com mock do OpenAI SDK:
     - LocalDesigner retorna entidades válidas
     - LocalDesigner retry após JSON inválido
     - LocalJudge retorna JudgeResult com score em [0,1]
     - LocalIterator filtra modification em entity com autor=user
     - Factory retorna Local* quando LLM_BACKEND=local
     - Erro claro se LOCAL_LLM_URL faltando

### Notas técnicas

- Qwen 2.5 Coder é tuned pra código. Pode ser mais "seco" em julgamento subjetivo estético. Se `judge_variety` ou `judge_cohesion` derem scores pouco discriminantes (tudo entre 0.4-0.6), reportar ao humano — pode ser sinal de trocar pro modelo general (Qwen3.6-35B-A3B disponível).
- Structured output via `response_format` é o caminho mais robusto pra esse modelo. Prompts devem ser explícitos sobre a estrutura esperada.
- Context atual do servidor: 16k tokens. Prompts do Iterator com histórico longo podem se aproximar — se estourar, avisar humano.
- llama-server aceita qualquer `model` no request. Usa o único carregado.
